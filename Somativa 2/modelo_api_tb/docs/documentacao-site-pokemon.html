1. Introdução

Este documento apresenta a documentação técnica e funcional do site de Pokemon desenvolvido em Laravel. O sistema tem como objetivo oferecer uma Pokedex web integrada à PokeAPI, permitindo consulta de informações oficiais, listagem com filtros, visualização detalhada de Pokemon, criação de novos registros personalizados e fusão de dois Pokemon em um novo cadastro.

A aplicação foi construída com foco acadêmico, demonstrando consumo de APIs externas, organização em padrão MVC, persistência em banco de dados MySQL, upload e armazenamento de imagens, uso de views Blade e integração opcional com inteligência artificial por meio da API Groq. Os jogos existentes no projeto não fazem parte do escopo desta documentação, conforme solicitado.

Objetivo Geral

Desenvolver uma aplicação web capaz de consultar, exibir, filtrar, criar e armazenar Pokemon, combinando dados externos da PokeAPI com dados próprios cadastrados no banco da aplicação.

Objetivos Específicos

  • Disponibilizar uma lista nacional de Pokemon com filtros por nome, número, geração, tipagem, origem e ordenação.
  • Permitir consulta detalhada de cada Pokemon em uma tela de Pokedex.
  • Possibilitar o cadastro manual ou assistido por IA de novos Pokemon.
  • Permitir fusão entre dois Pokemon, incluindo Pokemon oficiais e Pokemon criados pelo usuário.
  • Armazenar imagens enviadas pelo usuário e dados estruturados no banco MySQL.
  • Disponibilizar endpoints de API para busca, detalhes, criação, atualização e exclusão de Pokemon.

2. Situação Problema

A PokeAPI oferece uma grande quantidade de informações oficiais sobre Pokemon, porém o consumo direto desses dados pode ser pouco amigável para usuários finais e limitado para atividades que exigem criação de conteúdo próprio. Além disso, os Pokemon criados por usuários não existem na API oficial e precisam ser armazenados em uma base própria para que possam aparecer na lista, na Pokedex e nas operações de fusão.

Antes das melhorias implementadas, a aplicação dependia principalmente da consulta direta à API e não possuía uma experiência completa para listagem filtrada, persistência robusta de novos Pokemon em banco de dados, armazenamento de imagens enviadas pelo usuário e geração assistida por IA. Isso gerava dificuldade para manter dados personalizados disponíveis após recarregamento da aplicação ou em ambientes diferentes.

Problemas Identificados

  • Necessidade de reunir Pokemon oficiais e personalizados em uma mesma listagem.
  • Necessidade de filtrar Pokemon por critérios úteis, como nome, ID, geração, tipo e origem.
  • Necessidade de salvar Pokemon criados e fusões de forma permanente no banco de dados.
  • Necessidade de armazenar imagens enviadas pelo usuário e exibi-las na Pokedex.
  • Necessidade de reduzir o esforço de preenchimento manual por meio de geração assistida por IA.
  • Necessidade de manter consistência entre registros oficiais da PokeAPI e registros personalizados.

Solução Proposta

A solução adotada consiste em um site Laravel que consome dados da PokeAPI para Pokemon oficiais, armazena registros personalizados na tabela pokemons e unifica ambas as fontes na interface. A aplicação também utiliza cache em arquivo para consultas externas, upload para imagens locais e uma camada de repositório para padronizar a leitura e escrita dos Pokemon criados pelo usuário.

4. Metodologia

O desenvolvimento foi conduzido por análise incremental do código existente, identificação das rotas e views, criação de persistência em banco de dados e evolução das telas conforme os requisitos. O projeto utiliza a arquitetura MVC do Laravel, em que as rotas direcionam requisições para o controller, o controller aciona serviços e models, e as views Blade renderizam a interface.

Tecnologias Utilizadas

Tecnologia Uso no Projeto
LaravelFramework principal para rotas, controllers, validação, views, migrations, storage e cache.
PHPLinguagem utilizada no backend.
MySQLBanco de dados utilizado para persistir Pokemon personalizados.
BladeMotor de templates usado nas telas da aplicação.
Tailwind CSSEstilização responsiva das interfaces.
ViteBuild dos assets frontend do Laravel.
PokeAPIFonte externa de dados oficiais de Pokemon.
Groq APIServiço opcional de IA para geração de descrições e atributos.

Organização Técnica

  • routes/web.php: define as rotas web e endpoints internos.
  • PokemonController: concentra fluxos de listagem, Pokedex, criação, fusão e APIs internas.
  • CustomPokemonRepository: padroniza acesso aos Pokemon personalizados armazenados no banco.
  • Pokemon model: representa a tabela pokemons no Eloquent.
  • Migration pokemons: cria estrutura da tabela e importa dados antigos em JSON quando existirem.
  • Views Blade: renderizam lista, Pokedex, criação, escolha e fusão.

6.3.1. Modelo Conceitual

Conceitualmente, o sistema trabalha com uma entidade principal, Pokemon Personalizado, ligada a conjuntos de informações que descrevem suas características. Algumas dessas informações são armazenadas como listas ou objetos JSON para manter compatibilidade com a estrutura flexível da PokeAPI.

Pokemon Personalizado

  • Possui nome, número, origem e geração.
  • Possui altura, peso, experiência e descrição.
  • Possui uma ou mais tipagens.
  • Pode possuir imagem local ou URL externa.

Tipagem

  • Representa tipos como fire, water, grass e electric.
  • É usada para filtros, visualização e geração de golpes.
  • Cada tipo possui cor visual na listagem.

Ataque

  • Possui nome, tipo, categoria, poder, precisão, PP e efeito.
  • Pode ser informado manualmente ou gerado por IA.

Fusão

  • É resultado de dois Pokemon base diferentes.
  • Armazena os pais, origem da geração e payload da IA quando existir.
  • Também se comporta como um Pokemon na Pokedex.

Imagem

  • Pode ser upload enviado pelo usuário.
  • Pode ser URL externa.
  • Pode ser SVG gerado e sanitizado pelo sistema.

IA Payload

  • Representa os dados estruturados retornados pelo Groq.
  • Ajuda na auditoria do conteúdo gerado.

Relacionamentos Conceituais

  • Um Pokemon Personalizado possui muitas Tipagens, Habilidades, Estatísticas, Ataques, Evoluções e Variantes.
  • Uma Fusão é um Pokemon Personalizado que possui dois Pokemon base.
  • Uma Imagem pode estar associada a um Pokemon Personalizado.
  • Um IA Payload pode estar associado a Pokemon criados por IA ou fusões geradas por IA.

6.3.2. Modelo Lógico

No modelo lógico implementado, a flexibilidade dos dados foi concentrada em uma tabela principal chamada pokemons. Campos que naturalmente seriam tabelas relacionadas em uma normalização completa, como ataques e evoluções, foram armazenados como JSON para simplificar o desenvolvimento e preservar estruturas próximas às da PokeAPI.

Tabela: pokemons

Grupo Campos Descrição Lógica
Identificação id, pokedex_number, name, slug Controla identificação interna e identificação exibida ao usuário.
Classificação source, generation, rarity Define origem, geração e raridade do Pokemon.
Imagem image_path, image_url Permite usar arquivo local ou URL externa.
Dados básicos height, weight, base_experience, description Guarda informações principais exibidas na Pokedex.
Dados compostos types, abilities, stats, moves, evolutions, variants, locations Armazena listas e objetos em formato JSON.
Metadados meta, fusion, ai_payload Guarda dados extras, informações de fusão e retorno da IA.
Auditoria created_at, updated_at Registra datas de criação e atualização.

Índices e Restrições

  • pokedex_number unique: impede duplicidade de número na Pokedex.
  • index source/generation: melhora buscas por origem e geração.
  • index rarity: permite filtragens ou relatórios por raridade.

Justificativa do Uso de JSON

Como a PokeAPI fornece dados complexos e variados, o uso de JSON evita excesso de tabelas auxiliares e facilita salvar ataques, status, habilidades, evoluções e metadados gerados por IA. Para um projeto acadêmico e uma Pokedex customizável, essa abordagem reduz complexidade e mantém flexibilidade.

7. Conclusão

O site documentado cumpre o objetivo de funcionar como uma Pokedex personalizada, combinando dados oficiais da PokeAPI com registros criados pelo usuário. A aplicação permite consulta, listagem filtrada, visualização detalhada, criação manual, criação assistida por IA, fusão de Pokemon, upload de imagens e persistência em banco MySQL.

A estrutura em Laravel favorece organização do código e separação de responsabilidades. O uso de um repositório específico para Pokemon personalizados melhora a padronização dos dados exibidos nas views e reduz o acoplamento entre controller e banco. Já a integração com PokeAPI amplia o conteúdo disponível, enquanto o Groq adiciona uma camada criativa para geração automática de atributos.

Como evolução futura, sem considerar jogos, o sistema pode receber autenticação de usuários, painel administrativo, edição visual dos Pokemon cadastrados, versionamento das fusões, favoritos, exportação de dados e normalização adicional do banco em tabelas auxiliares para tipos, golpes e evoluções.

Resultado Final

  • Lista nacional com filtros e cards visuais.
  • Pokedex detalhada para Pokemon oficiais e personalizados.
  • Criação de Pokemon com dados completos ou auxílio do Groq.
  • Fusão entre Pokemon oficiais e criados pelo usuário.
  • Persistência de dados e imagens no ambiente da aplicação.
  • Documentação organizada conforme os tópicos solicitados.
